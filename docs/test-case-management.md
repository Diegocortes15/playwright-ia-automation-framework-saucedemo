# Test Case Management

## TL;DR

This project is **code-first**: test cases live in spec files, requests live in Jira tickets, results live in Playwright HTML reports. There is **no separate TCMS** (TestRail / Xray / Kiwi) by design — the existing pieces already cover what a code-first engineering team needs. Extension paths are documented below for when a client or audience requires more.

## Where things live

The chain from "we want this tested" to "here's the result":

| Layer                             | What lives here                                                                                                                                                    | Format                                   |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------- |
| **Test case request / spec**      | Jira ticket (project `SW`) — Feature + ACs in the description                                                                                                      | Ticket description                       |
| **Test case implementation**      | `tests/<feature>/<feature>.spec.ts`                                                                                                                                | TypeScript                               |
| **Request ↔ implementation link** | Provenance block at top of generated specs (`// Generated by /from-issue on YYYY-MM-DD from Jira SW-123`) + the GitHub-for-Jira app linking the PR onto the ticket | Comment block in spec file               |
| **Test case categorization**      | Nested describes (`Positive` / `Negative` / `Edge`) + `@smoke` tag in test title                                                                                   | TypeScript test annotations              |
| **AC traceability**               | AC coverage table in the PR body (`AC \| Test \| Bucket \| Status`)                                                                                                | Markdown table in PR                     |
| **Execution history**             | GitHub Actions workflow runs                                                                                                                                       | GH Actions UI                            |
| **Per-run rich report**           | Playwright HTML report (timeline, traces, screenshots, retries)                                                                                                    | `playwright-report/` (gitignored)        |
| **Machine-readable results**      | Playwright JSON report (for downstream tooling)                                                                                                                    | `test-results/results.json` (gitignored) |

## Reports

Already configured in `playwright.config.ts`:

```ts
reporter: [
  ['html', { open: 'never' }],         // interactive HTML, view with `npx playwright show-report`
  ['json', { outputFile: 'test-results/results.json' }],
  ['list'],                             // terminal output during runs
],
```

Common workflows:

- **View the last HTML report** (after a local run): `npx playwright show-report`
- **CI report viewing**: GitHub Actions uploads the `playwright-report/` directory as an artifact per Phase A.5
- **Downstream tooling** (e.g., a custom dashboard, Slack notification): parse `test-results/results.json`

What the HTML report shows that a TCMS typically charges for:

- Per-test timeline (which step failed, how long each step took)
- Step-level screenshots + DOM snapshots on failure
- Trace viewer (recorded interactions you can scrub through)
- Retry history per test
- Pass/fail counts per project (the 9-project matrix renders cleanly)

## Why no TCMS?

> **Update (ADR-0016, 2026-05-28):** TCMS remains **off by default** for the reasons below. An **opt-in one-way mirror** (default: Qase) is now available via `/from-issue` — see [`docs/tcms.md`](tcms.md) and [ADR-0016](adr/0016-tcms-mirror.md). The reasoning below is why it is opt-in, not on by default.

Test Case Management Systems (TestRail, Xray, Zephyr, Kiwi) shine when:

1. **Non-engineering stakeholders** (QA managers, PMs, auditors) need a UI to browse test cases and results without reading code
2. **Manual test cases** dominate the suite (TCMS gives QA testers a structured place to execute and record manual runs)
3. **Compliance / audit trails** require exportable evidence of what's tested vs. what's required
4. **Cross-team coordination** spans multiple repos / multiple test frameworks needing a unified view

For a **code-first Playwright project with engineering ownership**, TCMS often becomes net negative:

- **Duplicate maintenance**: every test rename or restructure needs sync between code and TCMS
- **ID drift**: TCMS test case IDs in code annotations rot when tests move or get refactored
- **Tooling overhead**: API token management, custom Playwright reporters to push results, ongoing schema sync
- **Second source of truth**: the code IS the test case; the TCMS becomes a stale copy

Modern Playwright/Cypress projects increasingly skip TCMS entirely. The patterns this framework uses (Jira tickets for requests, `/from-issue` for traceability, Playwright HTML for rich reporting) are the code-first equivalents — free, version-controlled, native to the development workflow.

## Extension paths

If your audience changes or a client requires more, here are the practical options:

### Rich pass/fail dashboards: Allure Report

[Allure](https://allurereport.org/) is free and integrates with Playwright via [`allure-playwright`](https://www.npmjs.com/package/allure-playwright). Adds:

- Severity / tag / feature filtering in a hosted dashboard
- Historical trend graphs
- Step-level breakdowns

Wire-up (~10 lines): add the package, register the reporter in `playwright.config.ts`, run `npx allure generate && npx allure open` after a test run. Worth it when a tech lead wants trends without paying for TestRail.

### Self-hosted TCMS: Kiwi TCMS

[Kiwi TCMS](https://kiwitcms.org/) is free open-source, Docker self-hosted, actively maintained. Real TCMS UI for QA teams who can't use git. Adds:

- Test plans, test cases, test runs, results in a web UI
- Manual + automated test tracking
- Requirements traceability

Wire-up (~1-2 days): Docker compose for Kiwi instance, a Playwright reporter that pushes results via the Kiwi REST API, conventions for mapping spec-file test names to Kiwi case IDs. Worth it when QA owns test cases and needs a UI.

### Commercial TCMS: TestRail / Xray / Zephyr

If a client mandates one of these, they each have established Playwright reporter plugins:

- TestRail: [`@playwright/test` + `testrail-api`](https://www.npmjs.com/package/@playwright/test-rail-reporter) or community reporters
- Xray (Jira): JUnit XML import (Playwright's `junit` reporter) + Xray's "Test Execution" issue type
- Zephyr Scale: similar JUnit-import workflow

These typically require: API token (env var), per-test annotation linking spec → TCMS case ID, a CI step to push results post-run. Plan ~1 day per integration.

## When to revisit

Reasons to add a TCMS (or richer reporting) later:

- A real client asks "where do we see test cases / results in a UI?"
- The team grows beyond engineers (QA testers, PMs, auditors need browse access)
- Regulatory / audit requirements demand exportable test evidence
- Manual testing becomes a significant share of the suite (Playwright handles automation; manual cases need somewhere to live)

Until one of those is concretely true, the current setup is sufficient.
