# Phase C.2.b — Positive / Negative / Edge Bucketing (Design Spec)

**Date:** 2026-05-20
**Status:** Draft — awaiting review
**Phase:** C.2.b (test bucketing). Builds on C.2.a (orchestrator skeleton). `@smoke` tag selection is C.2.c.

## 1. Goal & Scope

### Goal

Extend `/from-issue` (C.2.a) to organize generated tests into three nested describe blocks — **Positive**, **Negative**, **Edge** — based on LLM classification of each test. The PR description's AC coverage table gains a "Bucket" column so reviewers see the classification at a glance.

### What this IS

- A render-step + classification-step enhancement to the existing 13-step workflow. Step count unchanged; Steps 6 and 7 gain new behavior.
- A new reference doc (`bucket-classification.md`) that defines what each bucket means with worked examples.
- An AC coverage table expansion in the PR body (3 cols → 4 cols).

### What this IS NOT

- **Not a file split.** Still one generated file per issue. The bucketing is nested describes within a single outer describe.
- **Not new Playwright tags.** No `@positive`/`@negative`/`@edge` tags — tags continue to follow CLAUDE.md's existing taxonomy (`@no-auth`, `@standard`, etc.). The bucket is structural only.
- **Not `@smoke` tag selection.** That's still C.2.c.
- **Not retroactive.** Existing test files (login.spec.ts, cart/add-remove.spec.ts, etc.) stay flat; only NEW files generated by `/from-issue` get the bucket structure.
- **Not blocking on empty buckets.** If a feature has no Negative tests, the Negative describe is omitted entirely. Same for Edge.

## 2. Decision Log

1. **Bucket structure = nested describes within one file.** One outer `test.describe('<feature> (<auth-tag>)', ...)` containing up to three inner `test.describe('Positive' | 'Negative' | 'Edge', ...)` blocks. Not a file split, not comment-only separators.

2. **Bucket assignment = LLM judgment from AC text.** Same pattern as C.2.a's skip-signal (Decision 11). No BA syntax burden; classification is driven by the new `references/bucket-classification.md` reference doc.

3. **Empty buckets = omit entirely.** Only render describes with ≥1 test. A pure happy-path feature has only the `Positive` inner describe — no empty `Negative` or `Edge` blocks.

4. **Bucket labels = `Positive` / `Negative` / `Edge`.** Short conventional QA terminology. Reads cleanly in Playwright reports (e.g., `login (@no-auth) > Positive > @no-auth standard_user logs in successfully...`).

5. **Bucket order = fixed: Positive → Negative → Edge.** Even if Negative tests outnumber Positive, the file order stays Positive first. Predictability of structure over preservation of density.

6. **Bucket lives on the test, not the AC.** Because one test can cover multiple ACs (per C.2.a Decision 5), the LLM classifies the _test_ (based on its dominant behavior), not the underlying ACs. If a test spans a happy-path AC and an error-scenario AC, the LLM picks one bucket using `bucket-classification.md`'s ambiguity rules as the tiebreaker.

7. **PR description gains a Bucket column.** AC coverage table grows from 3 columns (`AC | Test | Status`) to 4 (`AC | Test | Bucket | Status`). Skipped ACs show `—` in the Bucket column. Generated tests show `Positive` / `Negative` / `Edge`.

8. **No single-bucket flag.** If all tests cluster into one bucket, no warning. Some features are genuinely single-bucket; trust the LLM and let reviewers push back via PR comment if needed.

9. **Zero automation-worthy ACs = no change from C.2.a.** Bucketing doesn't affect the 0-automatable abort path. Workflow Step 4 behavior unchanged. No PR opens, no file written, no buckets to worry about.

10. **Invalid bucket value = default to Edge + soft warning.** If the LLM emits `bucket: "Boundary"` or any value outside `{Positive, Negative, Edge}`, the orchestrator validates at the boundary between Step 6 and Step 7, defaults the test to `Edge` (the catch-all bucket), and records a soft warning in the PR body's Verification section: `⚠️ LLM emitted invalid bucket "<value>" for test "<title>" — defaulted to Edge. Reviewer: verify classification.`

11. **bucket-classification.md is the single source of truth.** No inline fallback rules. If the doc is missing or unreadable, abort with: _"`.claude/skills/from-issue/references/bucket-classification.md` not found. Re-install the skill or restore from git."_ Silent fallbacks hide drift.

```typescript
// Generated by /from-issue on YYYY-MM-DD from GitHub Issue #N.
// Source: <issue-url>
// Title: <issue-title>
```

### Extended AC analysis record (Step 6)

The internal record from C.2.a gains one field:

```
{
  id: 1,
  text: "<normalized AC text>",
  user: "<inferred saucedemo user>",
  worth_automating: true | false,
  rationale: "..." (only when worth_automating=false)
}
```

Per-test record (Step 6 output for grouped tests):

```
{
  title: "<behavior description>",
  covers: [1, 3],
  user: "<saucedemo user>",
  tags: ["<auth-tag>", "<user-tag-if-not-no-auth>"],
  bucket: "Positive" | "Negative" | "Edge"   ← NEW
}
```

### Bucket-classification.md responsibility

Single purpose: tell the LLM how to classify. Contains:

- One-paragraph definition per bucket
- 3-5 worked examples per bucket (concrete AC text → bucket assignment with rationale)
- 2-3 ambiguity rules ("perf assertion → Edge, not Negative"; "missing required field → Negative, not Edge")
- Cross-reference to test-template.md "Bucket structure" rule

Estimated ~60-80 lines. Mirrors the structural pattern of C.1's `component-detection.md`.

### Updated PR description AC coverage table

3 columns → 4 columns. New column inserted between Test and Status:

```markdown
## AC coverage

| AC                     | Test           | Bucket   | Status                  |
| ---------------------- | -------------- | -------- | ----------------------- |
| AC 1: <truncated text> | `<test title>` | Positive | ✅ generated            |
| AC 2: <truncated text> | `<test title>` | Negative | ✅ generated            |
| AC 3: <truncated text> | `<test title>` | Edge     | ✅ generated            |
| AC 4: <truncated text> | —              | —        | ⚠️ skipped: <rationale> |
```

Rules:

- Skipped ACs show `—` in the Bucket column (no test was generated, no bucket to report).
- Generated tests show exactly one of `Positive` / `Negative` / `Edge`.

## 4. Workflow Changes (Steps 6 + 7)

### Step 6 — AC analysis (extended)

**Before (C.2.a):** LLM groups automation-worthy ACs into a set of tests. Each test records `title`, `covers[]`, `user`, `tags[]`.

**After (C.2.b):** Same as before, plus each test record gains a `bucket` field. The LLM classifies the _test_ (not the AC) using `references/bucket-classification.md`.

Rationale for classifying on the test rather than the AC: one test may cover multiple ACs (per C.2.a Decision 5). If AC#1 is happy-path and AC#3 is an error scenario but the LLM groups them into a single integration test, that test goes to ONE bucket. The LLM picks the dominant behavior, with `bucket-classification.md`'s ambiguity rules as the tiebreaker.

Validation: before rendering (between Step 6 and Step 7), the orchestrator verifies each test's `bucket` is in `{Positive, Negative, Edge}`. On mismatch, default to `Edge` and record a soft warning for the PR body.

### Step 7 — Render test file (extended)

**Before:** One outer `test.describe('<feature> (<auth-tag>)', ...)` containing all `test(...)` calls flat.

**After:** Same outer describe. Inside, group tests by bucket and emit up to three nested describes in fixed order: `Positive` → `Negative` → `Edge`. Within each nested describe, tests appear in their Step 6 emission order.

Omit-empty rule: if a bucket has zero tests, don't emit its describe block at all. So a pure happy-path feature renders as one outer describe + one inner `Positive` describe (no Negative, no Edge).

### No changes to Steps 1-5, 8-13

The `bucket` field flows through Step 7 into the rendered file, and through Step 12 into the PR body's AC coverage table. No other step needs to change:

- Step 8 (write file) — same path, same overwrite refusal
- Step 9 (typecheck) — nested describes typecheck identically to flat
- Step 10 (test run) — Playwright reports show buckets in test paths; `--reporter=list` output is unaffected
- Step 11 (commit + push) — unchanged
- Step 12 (PR body) — adopts the 4-column AC coverage table from the updated template
- Step 13 (issue comment + report) — unchanged

## 5. Failure Modes & Edge Cases

### Inherited from C.2.a, unchanged

All 11 failure modes from C.2.a §4 carry over identically — bucketing doesn't change abort behavior. In particular:

- **Issue body free-form** → abort with "couldn't extract ACs"
- **Zero ACs automation-worthy** → abort + issue comment (no PR, no file, no buckets)
- **Page Name collision** → reuse + flag (orthogonal to bucketing)
- **Typecheck fails / test run fails** → open PR anyway with failures surfaced

### New for C.2.b

| Failure                                                                                          | Behavior                                                                                                                                                                                                                                                                                                                                          |
| ------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| LLM emits an invalid bucket value (e.g., `"Boundary"` instead of `"Positive"/"Negative"/"Edge"`) | Orchestrator validates each test record's `bucket` field against the closed set `{Positive, Negative, Edge}` before rendering. On mismatch: default to `Edge` and record a soft warning in the PR body's Verification section: `⚠️ LLM emitted invalid bucket "<value>" for test "<title>" — defaulted to Edge. Reviewer: verify classification.` |
| `bucket-classification.md` missing/unreadable                                                    | Abort with: _"`.claude/skills/from-issue/references/bucket-classification.md` not found. Re-install the skill or restore from git."_ Don't fall back to inline rules — the doc is the source of truth and silent fallbacks hide drift.                                                                                                            |
| All tests cluster into one bucket                                                                | No failure. Per Decision 8 (no single-bucket flag), this is acceptable. Reviewer sees the classification in the AC coverage table and can push back via PR comment if needed.                                                                                                                                                                     |

### Edge case worth naming explicitly

**Test ordering inside the file diverges from AC emission order.** Step 6 may emit tests as AC1 → AC2 → AC3, but Step 7 reorders them into Positive → Negative → Edge buckets. A reviewer reading the file top-to-bottom won't find tests in AC order. This is intentional — predictability of structure wins over preservation of AC order. The AC coverage table in the PR body provides AC→test traceability for anyone who needs that view.

### Two principles

1. **Validate at the boundary** (between Step 6 and Step 7) — bucket values must be in the closed set before any file is written. Caught at the source.
2. **Default to Edge on classification ambiguity, not Positive.** "Edge" is semantically the catch-all bucket. Defaulting to Positive would silently hide misclassification; defaulting to Edge surfaces it.

## 6. Verifying the Skill Works

### Primary smoke test (one-time, run when C.2.b ships)

Craft a `to-be-automated` test issue with ACs spanning all three buckets:

> **Feature:** login
> **Page Name:** LoginPage
> **AC 1:** Standard user logs in with valid credentials and lands on inventory. _(→ Positive)_
> **AC 2:** Locked-out user sees the lockout error message. _(→ Negative)_
> **AC 3:** Invalid password shows a generic error message. _(→ Negative)_
> **AC 4:** Username with leading/trailing whitespace is rejected the same way as malformed input. _(→ Edge)_

Run `/from-issue <num>`. Expected:

1. Skill executes the 13-step workflow (step count unchanged from C.2.a).
2. LoginPage reused (collision warning surfaces in PR — same as C.2.a).
3. LLM produces 4 tests, classified as: 1× Positive, 2× Negative, 1× Edge.
4. Generated file has the outer `test.describe('login (@no-auth)', ...)` containing three nested describes: `Positive` (1 test), `Negative` (2 tests), `Edge` (1 test) — in fixed order.
5. PR body's AC coverage table has 4 columns including `Bucket`, with the LLM's classification visible per row.
6. Typecheck ✅, test run ✅ all 4 pass.

After verifying: close PR, delete branches, close issue (same cleanup as C.2.a Task 8).

### Secondary smoke test (omit-empty behavior)

Optional second smoke run with a pure-Positive issue (e.g., "AC 1: User can view inventory after login. AC 2: User can sort products by price."). Expected: file contains only the `Positive` inner describe — no empty `Negative` or `Edge` blocks.

### What we deliberately do NOT test as part of C.2.b delivery

- LLM classification accuracy at scale (it's a judgment call; one smoke pass is enough to verify the mechanism works).
- Invalid-bucket fallback (synthetic; would require mocking the LLM output).
- bucket-classification.md missing (synthetic; would require deleting the file).

## 7. Deferred + Open Questions

### Explicitly deferred

| Concern                                                    | Phase     |
| ---------------------------------------------------------- | --------- |
| `@smoke` tag applied to a subset of tests                  | **C.2.c** |
| Label catalog (other labels route to different sub-skills) | **C.3**   |
| Conditional sub-agent dispatch by label                    | **C.4**   |

### Open questions (not blocking C.2.b)

1. **Classification stability across LLM model versions.** A future Opus/Sonnet might bucket the same ACs differently. No automated guard. Mitigation: PR review gate catches drift; if classifications consistently feel "off", update `bucket-classification.md` with sharper examples.
2. **Per-bucket tag filtering.** Should we add `@positive`/`@negative`/`@edge` tags so reviewers can run `npx playwright test --grep @negative`? Out of scope for C.2.b — but worth revisiting if reviewers ask for it.
3. **Cross-bucket test ordering for AC traceability.** Reviewers wanting to read tests in AC order have to consult the PR coverage table. If this becomes painful in practice, consider adding `// covers: AC1, AC3` comments above each test (lightweight) or splitting into per-AC files (heavyweight, conflicts with Decision 1).

## Summary

C.2.b adds Positive/Negative/Edge bucket headers to tests generated by `/from-issue`. Implementation is a focused extension of C.2.a's workflow: Step 6 classifies each test via LLM judgment driven by a new `bucket-classification.md` reference doc, Step 7 renders nested describes in fixed Positive→Negative→Edge order with empty buckets omitted, and the PR body's AC coverage table gains a `Bucket` column. Six files touched (1 new + 5 modified), zero changes outside the skill directory.

C.2.c (`@smoke` tag selection) is the next phase in the arc.
