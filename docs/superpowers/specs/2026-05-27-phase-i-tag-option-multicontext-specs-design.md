# Phase I — Playwright `tag` Option for Multi-Context Feature Specs — Design

**Status:** Approved (brainstorm) — pending implementation plan.
**Date:** 2026-05-27

## Goal

Let one feature spec file hold tests for **multiple user/auth contexts** by moving every tag from the describe/test **title string** into Playwright's native **`{ tag }` option**. This dissolves the collision class SW-4 exposed and makes `/from-issue` handle "same feature, different user context" flawlessly — reusably on a brand-new project.

## Context

SW-4 (`@standard` inventory content) couldn't be generated because `tests/inventory/inventory.spec.ts` already exists as SW-3's `@problem` spec, and the framework's convention bakes the routing tag into the describe **title** (`test.describe('inventory @problem', …)`) with **one outer describe per file**. A second user context has nowhere to live. This is a self-imposed constraint: the wider Playwright idiom separates **behavior** (file + describe structure) from **context** (user/role/browser → tags + projects + fixtures), and Playwright ≥1.42 (repo is on **1.59.1**) provides a first-class **`{ tag }` option** so tags need not be in the title.

**Verified linchpin (empirical):** project `grep` matches tags set via the `{ tag }` option, not just titles. A throwaway probe with `{ tag: '@standard' }` / `{ tag: '@problem' }` describes confirmed `--project=standard` selects only the `@standard` describe, `--project=problem` only the `@problem` one, and `--project=no-auth` neither. So routing keeps working with zero project-config changes.

## Decisions

1. **All tags move to the `{ tag }` option.** Routing tags (`@no-auth` / `@all-users` / `@standard` / `@problem` / `@performance_glitch` / `@error` / `@visual`) go on the **describe**; `@smoke` goes on the **test**. Titles become pure prose. The Phase-F "no parentheses around the auth-tag in the title" rule is deleted (moot).

2. **One file per feature; sibling tagged describes per user-context.** `tests/<feature>/<feature>.spec.ts` holds one `test.describe('<feature> — <context-label>', { tag: '<routing-tag>' }, …)` per context, side by side. Bucket describes (Positive/Negative/Edge) stay **inside** each context describe. Nesting is describe → describe → test.

3. **Context-label scheme.** The describe title is `'<feature> — <context-label>'`, where the label maps the routing tag to a readable name: `@standard`→`standard_user`, `@problem`→`problem_user`, `@performance_glitch`→`performance_glitch_user`, `@error`→`error_user`, `@visual`→`visual_user`, `@no-auth`→`no auth`, `@all-users`→`all users`.

4. **Test titles are pure prose**; smoke tests carry `{ tag: '@smoke' }` on the `test()` call. A smoke test inside a context describe is effectively tagged `[<routing-tag>, @smoke]` (Playwright merges describe + test tags) — so both `--project=<user>` and `--grep @smoke` select it.

5. **Augment match key = the routing tag.** When `/from-issue` augments a feature file: find the sibling describe whose `{ tag }` matches the new tests' context. **Match → insert** into its bucket describes. **No match → add a new sibling tagged describe** (with bucket children) in the same file. The old "auth-tag must match or abort" pre-check is **removed** — it was the blocker. `--new-file` stays as an escape hatch; the "structure unrecognizable" abort stays. `// Generated by` + `// Augmented by:` contributor tracking stays at the file level; re-running a contributing `<KEY>` still refuses.

6. **Retrofit the 3 clean-room specs** on the `e2e` line — `login` (`@no-auth`), `footer` (`@standard`), `inventory` (`@problem`) — to the tag-option shape; the full suite must stay green. **`main`'s full original suite is out of scope** (it still routes — grep matches titles too; migrating it is a separate optional follow-up), mirroring how Phase H left `main`'s config alone.

7. **ADR-0015** records the convention and supersedes the tag-in-title rule in `test-template.md`.

## Spec shape (canonical)

```ts
// tests/inventory/inventory.spec.ts
test.describe('inventory — problem_user', { tag: '@problem' }, () => {
  test.describe('Edge', () => {
    test('every product image is the same broken placeholder', async ({ inventoryPage }) => {
      /* … */
    });
  });
});

test.describe('inventory — standard_user', { tag: '@standard' }, () => {
  test.describe('Positive', () => {
    test('all six product titles render exactly', async ({ inventoryPage }) => {
      /* … */
    });
  });
  test.describe('Edge', () => {
    test('product title turns green on hover', async ({ inventoryPage }) => {
      /* … */
    });
  });
});
```

## `/from-issue` workflow changes

- **Step 7 (render):** emit `test.describe('<feature> — <context-label>', { tag: '<routing-tag>' }, …)` per context; bucket describes inside; test titles prose; `{ tag: '@smoke' }` on smoke tests. No tags in any title.
- **Step 8 (collision table):** unchanged for the refuse-on-same-`<KEY>` and `--new-file` rows. The "file exists, `<KEY>` not a contributor → AUGMENT" row now always routes to Step 8.5 (no auth-tag gate).
- **Step 8.5 (augment):** replace the "auth-tag must match or abort" pre-check with **find-or-create by routing tag**: locate the sibling describe with the matching `{ tag }`; insert into its buckets if present, else add a new sibling tagged describe. Keep the duplicate-guard and "structure unrecognizable" abort.

## Affected files (implementation surface)

- `.claude/skills/from-issue/references/test-template.md` — the canonical shape (tag option, sibling context describes, prose titles); delete the parens rule.
- `.claude/skills/from-issue/references/workflow.md` — Steps 7, 8, 8.5.
- `.claude/skills/from-issue/references/pr-description-template.md` — prose titles; `⚡` smoke marker keys off the `@smoke` tag, not the title.
- `docs/adr/0015-spec-tags-via-tag-option.md` (new).
- `CLAUDE.md` — tag-conventions table note (tags applied via the `{ tag }` option; routing unchanged).
- Retrofit (e2e line): `tests/login/login.spec.ts`, `tests/footer/footer.spec.ts`, `tests/inventory/inventory.spec.ts`.

## Alternatives considered

- **Split by user into separate files** (`inventory-SW-4.spec.ts`, i.e. `--new-file`). Rejected as the primary fix: user/role is a matrix dimension, not a file boundary; fragments a feature's coverage across files. Kept only as a manual escape hatch.
- **Keep tag-in-title, allow multiple outer describes per file.** Rejected: still pollutes titles, and the report chips/duplicate-tag issues that motivated the Phase-F rule persist; the `{ tag }` option is the purpose-built mechanism.
- **Move only the routing tag, keep `@smoke` in the title.** Rejected (user decision): mixed convention, dual logic in templates, future foot-guns.

## Scope / non-goals (YAGNI)

- No change to bucket or smoke **policy**, the project matrix, or Phase H harness growth.
- No retrofit of `main`'s full original suite.
- No new tags — same vocabulary, applied via the option.
- No change to fixtures or Page Object conventions.

## Verification approach

1. **Probe (done):** project grep matches option-tags. ✅
2. **Retrofit green:** after converting the 3 specs, `npm test` on `e2e` passes with the same test count, each routed to its project (`@no-auth`→no-auth, `@standard`→standard, `@problem`→problem).
3. **SW-4 re-run (post-merge, user-driven):** `/from-issue SW-4` augments `tests/inventory/inventory.spec.ts` by adding a `@standard` sibling describe next to SW-3's `@problem` one — no collision, no `--new-file`, suite green.
4. **Smoke routing:** `npm run test:smoke` (`--grep @smoke`) still selects smoke tests now that `@smoke` is an option tag (grep matches option-tags — proved).
