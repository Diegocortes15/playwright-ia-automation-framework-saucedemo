# Phase E — Jira as the Ticket Source for `/from-issue` — Design

**Status:** Approved (brainstorm) — pending implementation plan.
**Date:** 2026-05-24

## Goal

Retarget `/from-issue` to read its source ticket from **Jira** (project key `SW`, tickets `SW-1`, `SW-2`, …) instead of GitHub Issues, so the saucedemo template represents how real QA orgs drive automation. Jira **fully replaces** GitHub Issues as the ticket source. GitHub remains the code host (the skill still opens a PR there).

## Context / problem

`/from-issue` is GitHub-coupled end to end: it fetches via `gh issue view`, gates on the `to-be-automated` label, keys off the issue number, opens a PR, and comments the PR link back on the issue. The analysis/rendering pipeline behind it (Steps 4–10) is already source-agnostic — the references `test-template.md`, `bucket-classification.md`, `smoke-policy.md`, `qa-analysis.md`, `playwright-conventions.md`, `data-placement.md` say nothing about GitHub. So retargeting touches only the **ends** of the workflow (fetch + write-back), not the middle.

The earlier project premise ("GitHub Issues for tickets; no Jira access") is now obsolete — the user created a Jira project. This reverses that premise (recorded in memory and the C2a skeleton design), captured in a new **ADR-0011**.

## Decisions

1. **Jira fully replaces GitHub Issues** as the ticket source. The old GitHub-Issues read path is retired, not kept alongside.

2. **Keep the skill named `/from-issue`**, retargeted to Jira. Jira's own data model calls tickets "issues", so the name stays accurate, the skill identity and all six analysis/rendering references are untouched, and only the fetch + write-back steps change. Invocation: `/from-issue SW-123`.

3. **No readiness gate.** The old Step 3 (`verify to-be-automated label`) is removed. The operator explicitly invoking `/from-issue SW-123` is the intent signal. The existing Step 4 backstop ("no ACs worth automating → abort") still fails safe on a non-spec ticket.

4. **Read access via the Atlassian MCP (OAuth).** The skill reads the ticket through the Atlassian MCP's get-issue tool — no `.env`, no API token, no local Jira secret. This is a **scoped** adoption of MCP for Jira; it does not blanket-reverse [ADR-0007](../../adr/0007-gh-cli-not-github-mcp.md) (`gh` CLI stays for GitHub). Rationale: the Atlassian MCP is the official token-free OAuth path and there is no first-class Jira CLI as ubiquitous as `gh`.

5. **No Jira write-back.** The user has **GitHub for Jira** connected, so emitting the issue key in the branch name + PR title is sufficient — the app auto-shows the PR in the ticket's Development panel. The old Step 13 (comment-back) is removed, and the skill needs **zero Jira write access**. _Fallback:_ if the first PR does not auto-link, the app isn't wired to this repo's org and we add a comment-back step then.

6. **Provenance & augment (Phase D.2) carry over, keyed by Jira key.** Header `// Generated by /from-issue on <date> from Jira SW-123.` and `// Augmented by: SW-456 (date)`. The contributor set is the set of Jira keys; re-running a contributing key refuses (per ADR-0010).

## Access summary

| Concern           | Mechanism                                                     |
| ----------------- | ------------------------------------------------------------- |
| Read ticket       | Atlassian MCP (OAuth) — no env vars                           |
| Open PR + branch  | `gh` / `git` (unchanged)                                      |
| Link PR → ticket  | GitHub-for-Jira app, via key in branch + PR title (automatic) |
| Jira write access | none required                                                 |

The skill needs no Jira secret of any kind.

## Workflow changes

Only the ends change; analysis Steps 4–10 are reused verbatim.

- **Step 1 — Validate.** The arg is a Jira **issue key** matching `^[A-Z][A-Z0-9]+-\d+$` (e.g. `SW-123`); if missing/malformed, ask — don't guess. Verify the Atlassian MCP is reachable (a get-issue/whoami probe); if not, abort: _"The Atlassian MCP isn't connected. Connect it in Claude Code (OAuth) and re-run."_
- **Step 2 — Fetch.** Read the ticket via the Atlassian MCP's get-issue tool for key `<KEY>`. Capture `summary` and the `description` (rendered/text form). On not-found / no-access, abort with the MCP error verbatim. (Exact MCP tool name confirmed once the MCP is connected; the workflow references it generically until then.)
- **Step 3 — Gate.** **Removed.**
- **Steps 4–10 — Unchanged.** Parse Feature + ACs from summary + description (free-form / GWT supported, exactly as today's body parsing), QA-analysis (merge/split/skip), bucket + smoke classification, render (incl. D.2 augment-vs-create mode resolution), isolated typecheck, run.
- **Step 11 — Branch/commit.** Branch `from-issue/<KEY>-<feature>` (e.g. `from-issue/SW-123-login`). Commit message via repeated `-m` flags (no here-strings, per phase-d1.5). Filename stays `<feature>.spec.ts` (phase-d1.2).
- **Step 12 — PR.** Title `feat: tests from <KEY> — <summary>` (truncate summary to keep the title ≤ ~60 chars after the key); reference `<KEY>` in the PR body so the GitHub-for-Jira app links it.
- **Step 13 — Write-back.** **Removed** (app auto-links). No Jira comment.

## ADR-0011

New ADR: **Jira as the ticket source for `/from-issue`, read via the Atlassian MCP.** Records: Jira fully replaces GitHub Issues; read via Atlassian MCP (OAuth, token-free); write-back delegated to the GitHub-for-Jira app (no Jira write access); the gate is dropped. **Scopes** ADR-0007 rather than reversing it — `gh` CLI remains for GitHub; the Atlassian MCP is adopted only for Jira, justified by token-free OAuth and the absence of a first-class Jira CLI. Marks ADR-0007 as "amended by ADR-0011". Reverses the "GitHub Issues / no Jira" premise from the C2a skeleton design.

## Scope / non-goals (YAGNI)

- No status transitions, no JQL/bulk listing, no custom "Acceptance Criteria" field (ACs live in the description).
- No Jira write access (no comment, no field updates) — the app owns linking.
- `gh` still opens the PR; one feature file per ticket; multi-page issues handled as today.
- No env-var/token path (MCP only). No support for both GitHub + Jira (full switch).

## Open prerequisites (block implementation/testing, not design)

1. **Connect the Atlassian MCP** to Claude Code (OAuth). It is not connected in the current session — no Jira MCP tools are available yet.
2. **Create a test ticket** in project `SW` (e.g. recreate login as `SW-1`) with Feature + ACs in the description.
3. Confirm **GitHub for Jira** is connected to this repo's GitHub org + the Jira site (the first PR's auto-link is the verification).

## Verification approach

After the prerequisites: on the blank-slate branch, run `/from-issue SW-1` (login) → confirm the MCP fetch returns the ticket, Feature + ACs parse, tests generate + run, a PR opens on GitHub with `SW-1` in the branch + title, and the PR auto-appears on the `SW-1` ticket's Development panel. Then a second `SW` ticket extending login exercises D.2 augment, keyed by Jira key.
