# Architecture

## Overview

This is a Playwright + TypeScript test framework targeting [saucedemo.com](https://www.saucedemo.com). It covers the six user personas that saucedemo exposes (`standard`, `problem`, `performance_glitch`, `error`, `visual`, and `locked_out`), exercises every major UI flow (login, inventory browsing, sorting, cart, checkout), and enforces strict typing, Playwright-aware lint rules, and consistent code style.

The framework sits squarely in the **code-first, AI-assisted** camp вЂ” the industry standard pattern in 2026. Tests remain fast, deterministic, debuggable, and version-controlled. The framework is conventional Playwright; AI agents will _write_ the code rather than executing tests at runtime. Pure AI-driven runtime test execution (Mabl, Playwright-MCP-only setups) was explicitly rejected because it is too slow and too flaky for regression suites.

Phase C+ will layer on the `/from-jira` orchestrator, a code-review skill, and sub-agents that read tickets and author PRs. Those agents use the docs in this directory and `CLAUDE.md` as their primary context. Every structural and naming decision in the framework was made with AI readability in mind: straight-line code, no fluent navigation chains, typed loaders with predictable names, explicit fixture injection.

---

## Tech Stack

- **Node.js 22.x** вЂ” runtime; required for native ESM import attributes (`with { type: 'json' }`)
- **TypeScript 5.9** (`strict: true`, `noUnusedLocals`, `noUnusedParameters`) вЂ” language; strict mode keeps AI-generated code correct over time
- **Playwright 1.59.x** (chromium + firefox + webkit) вЂ” test runner and browser automation engine
- **ESLint v9 flat config + `eslint-plugin-playwright`** вЂ” lint with Playwright-specific safety rules (`no-wait-for-timeout`, `prefer-web-first-assertions`, etc.) that Biome does not yet provide
- **Prettier 3** (with `endOfLine: 'auto'` for cross-platform CRLF/LF compatibility) вЂ” formatter
- **dotenv** вЂ” local env var loading (`.env` в†’ `src/utils/env.ts` single read point)
- **GitHub Actions on `ubuntu-latest`** вЂ” CI; single job running the full 9-project matrix

### Directory guide

**`src/pages/`** вЂ” Page object classes, one per saucedemo route. Each page composes shared Components and holds its own page-unique Locators. The `checkout/` subdirectory contains the three-step checkout flow (`CheckoutInfoPage`, `CheckoutOverviewPage`, `CheckoutCompletePage`).

**`src/components/`** вЂ” Reusable UI building blocks: `Header`, `CartBadge`, `ProductCard`, and `SortDropdown`. Components scope all their locators to a `root` Locator and never hold any reference to a parent page or sibling component.

**`src/fixtures/test.ts`** вЂ” Custom Playwright `test` fixture that pre-instantiates all page objects and re-exports `expect`. Every spec file imports `test` and `expect` from `@fixtures/test` вЂ” never from `@playwright/test` directly.

**`src/utils/`** вЂ” `env.ts` provides a typed `EnvConfig` object as the single read point for `process.env`; `logger.ts` is a level-gated logger stub used by pages and components.

**`data/types.ts`** вЂ” TypeScript interfaces for all test data shapes: `Product`, `CheckoutScenario`, and `SortOption`. These are the single source of truth for data structure across the entire codebase.

**`data/fixtures.ts`** вЂ” Typed loader functions that import JSON via ESM import attributes; one loader per JSON source following the `load<Subject>(): <Subject>[]` naming convention. Also contains the `getProductById(id)` helper.

**`data/shared/`** вЂ” Reference data shared across multiple feature areas (e.g., `products.json` used by both inventory and checkout tests).

**`data/scenarios/<feature>/`** вЂ” Parameterized test scenarios scoped to a specific feature: `checkout/valid-checkout.json`, `checkout/invalid-postalcode.json`, `sort/sort-orders.json`.

**`tests/auth.setup.ts`** вЂ” Logs in each of the five supported users at the start of every test run and saves `storageState` to `auth/<user>.json`.

**`tests/<feature>/*.spec.ts`** вЂ” Spec files organized by feature area: `login/`, `inventory/`, `cart/`, `checkout/`, `visual/`.

**`auth/`** вЂ” Generated storageState files (git-ignored; `.gitkeep` is tracked so the folder exists on a clean clone).

**`docs/`** вЂ” This directory. Architecture, ADRs, app behavior docs, and design specs live here.

**`playwright.config.ts`** вЂ” Defines 9 Playwright projects: `setup`, `no-auth`, 5 chromium user projects, `firefox-standard`, `webkit-standard`.

**`.github/workflows/test.yml`** вЂ” GitHub Actions CI workflow; runs the full 9-project matrix on every push and pull request to `main`.

## Composition Rules

These rules are the contract every contributor (human or AI agent) must follow when extending the framework. They exist to keep code readable, components reusable, and AI-generated extensions consistent.

1. **Component knows about Locators and (optionally) child Components only.** A component never imports or references a Page, and it never knows about its parent. This keeps components context-free so they can be composed into any page without modification.

2. **Page composes Components and holds page-unique Locators.** The page is the composition root. It wires components together and adds any locators that belong only to that route вЂ” but it never composes other Pages or holds cross-page references.

3. **Pages NEVER return other Pages.** Page action methods return `void` or data. Tests hold all page fixtures via injection and navigate explicitly. Fluent navigation chains (e.g., `loginPage.login().goto()`) were explicitly rejected because they create cross-page import coupling and make code harder for AI agents to read.

4. **Tests know about Pages and Data only.** A test body never reaches into a raw `Locator` or constructs a `Component` directly. This boundary ensures tests read like user intent rather than implementation detail.

5. **All locator/component fields are `readonly`.** Set once in the constructor, never reassigned. Mutability in page objects is a source of subtle bugs in parallel test runs.

6. **Constructor order: composed Components first, then page-direct Locators.** This convention makes it immediately obvious which pieces are shared abstractions and which are local to the page. AI agents extending a page can follow the pattern without reading surrounding context.

7. **Action methods read like English.** Method names such as `addProductToCart`, `sortBy`, and `fillForm` make tests near-prose. A reader unfamiliar with the codebase should be able to understand a test's intent at a glance.

8. **Queries return data, never `Locator`.** A method like `getProductNames()` returns `string[]`, not a `Locator` array. Keeping `Locator` objects inside page/component boundaries prevents test code from depending on Playwright internals.

9. **Components scoped to one of many similar elements take a discriminator in the constructor.** For example, `new ProductCard(page, productName)` anchors all internal locators under a `root` filtered by `hasText: productName`. The discriminator is always passed at construction time, never mutated later.

10. **Refactor a page-direct locator into a Component the moment a second page needs it.** Don't wait for a third page. Duplication across pages is a maintenance trap and breaks the principle that locator strings live in exactly one place.

11. **Component nesting depth is at most 2.** `Header` may compose `CartBadge` (depth 2). A component that needs to compose a component that composes another component indicates a design problem вЂ” flatten or redesign.

12. **No `await page.waitForTimeout()` ever.** Use Playwright's auto-waiting assertions (`expect(...).toBeVisible()`, `expect(...).toHaveURL()`, etc.). Arbitrary timeouts are a flakiness source and are blocked by the `playwright/no-wait-for-timeout` lint rule.

### Selector preference order

When choosing how to identify a UI element, prefer this order:

1. `[data-test="..."]` attribute (most stable; saucedemo provides these widely)
2. `getByRole(...)` with anchored regex (e.g., `/^Add to cart$/i`)
3. Text matchers (`getByText`, `hasText` filters)
4. CSS selectors вЂ” only when none of the above is available

**Never use XPath.** This is the framework's only hard rule about selectors. The Phase A code-review process found XPath selectors brittle, hard to read, and incompatible with Playwright's auto-waiting semantics.

The only legitimate non-`data-test` locator currently in the framework is `LoginPage.errorCloseButton` (the `.error-button` CSS class) вЂ” saucedemo doesn't expose a `data-test` attribute for that element.

For the historical decision behind this pattern, see [ADR-0001](adr/0001-pom-by-component.md).

## Data Layer

Test data is split between two directories under `data/`: reference data shared across features (`data/shared/`) and parameterized scenario data scoped to a single feature (`data/scenarios/<feature>/`). For example, `data/shared/products.json` is the canonical product catalog used by both inventory and checkout tests, while `data/scenarios/checkout/valid-checkout.json` lists only the checkout-specific scenario inputs.

**`data/types.ts`** is the single source of truth for data structure. The three interfaces вЂ” `Product`, `CheckoutScenario`, and `SortOption` вЂ” define exactly what shape each JSON file must conform to. Adding a field to the JSON without updating the interface produces a TypeScript error at the load site, not a silent runtime failure.

Loaders in **`data/fixtures.ts`** follow the convention `load<Subject>(): <Subject>[]` вЂ” one loader per JSON source. Examples: `loadProducts()`, `loadValidCheckouts()`, `loadSortOrders()`. AI agents extending the framework match this pattern when adding new scenario files. Tests import loaders via the `@data/*` path alias (`import { loadSortOrders } from '@data/fixtures'`).

The **`getProductById(id: string): Product`** helper in `data/fixtures.ts` throws a descriptive error when the requested id is absent. Checkout tests use this helper to resolve a product by id without scattering find-and-throw logic across test bodies.

JSON imports use **ESM import attributes** (`import x from './x.json' with { type: 'json' }`). This replaced the earlier `createRequire` workaround that was a CommonJS escape hatch in an ESM file вЂ” a convention-drift trap for AI agents extending `data/fixtures.ts`. The import-attributes form is natively supported in Node 22.x + TypeScript 5.9 and is the pattern all new loaders must follow.

See [ADR-0003](adr/0003-data-hybrid-shared-scenarios.md) for the layout rationale and [ADR-0005](adr/0005-esm-import-attributes-for-json.md) for the JSON-import decision.

## Multi-User Infrastructure

### Auth setup

`tests/auth.setup.ts` logs each of the five supported users in via the UI at the start of every test run, then persists `storageState` to `auth/<user>.json`. The five users are `standard`, `problem`, `performance_glitch`, `error`, and `visual`. `locked_out_user` is deliberately excluded вЂ” login fails for that user, so there is no session to save. The locked-out user is exercised only in `@no-auth` login tests where the negative-path behavior is the point.

StorageState files are engine-portable for saucedemo's basic cookie-based session, so `auth/standard.json` is reused by all three browser engines (`chromium`, `firefox`, `webkit`) without a per-browser auth setup step.

### Playwright projects (9 total)

`playwright.config.ts` defines nine projects:

- **`setup`** вЂ” runs `tests/auth.setup.ts`; all authenticated projects depend on it
- **`no-auth`** вЂ” chromium, no storageState; grep `@no-auth`; for login/logout tests
- **`standard`** вЂ” chromium; grep `@all-users|@standard|@sort-functional`; depends on setup
- **`problem`** вЂ” chromium; grep `@all-users|@problem`; depends on setup
- **`performance_glitch`** вЂ” chromium; grep `@all-users|@performance_glitch|@sort-functional`; `navigationTimeout: 30_000` override (saucedemo deliberately delays this user ~10 s; the default 15 s leaves no headroom); depends on setup
- **`error`** вЂ” chromium; grep `@all-users|@error`; depends on setup
- **`visual`** вЂ” chromium; grep `@all-users|@visual|@sort-functional`; depends on setup
- **`firefox-standard`** вЂ” Desktop Firefox; grep `@all-users|@standard|@sort-functional`; storageState `auth/standard.json`; depends on setup
- **`webkit-standard`** вЂ” Desktop Safari; grep `@all-users|@standard|@sort-functional`; storageState `auth/standard.json`; depends on setup

### Tag conventions

| Tag                   | Runs on project(s)                                                                | Purpose                                                                                           |
| --------------------- | --------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| `@no-auth`            | `no-auth`                                                                         | Login/logout tests, no pre-existing session                                                       |
| `@all-users`          | All 5 chromium user projects + firefox/webkit                                     | User-agnostic flows                                                                               |
| `@standard`           | `standard`, `firefox-standard`, `webkit-standard`                                 | Tests where only standard user is meaningful                                                      |
| `@problem`            | `problem`                                                                         | Tests that _expect_ the problem user's broken UI                                                  |
| `@performance_glitch` | `performance_glitch`                                                              | Tests that handle slow loads                                                                      |
| `@error`              | `error`                                                                           | Tests for the error user's random failures                                                        |
| `@visual`             | `visual`                                                                          | Visual regression for the visual user                                                             |
| `@sort-functional`    | `standard`, `performance_glitch`, `visual`, `firefox-standard`, `webkit-standard` | Sort tests (excluded from `problem`/`error` вЂ” saucedemo breaks the sort dropdown for those users) |

`@sort-functional` exists because saucedemo intentionally breaks the sort dropdown for `problem_user` and `error_user`. Tagging sort tests with `@sort-functional` instead of `@all-users` automatically opts in the three users whose sort works (`standard`, `performance_glitch`, `visual`) plus the two cross-browser projects, while keeping `problem` and `error` out.

### Cross-browser smoke pattern

Chromium runs the full 5-user matrix. Firefox and webkit run only the standard user. This catches real browser rendering and locator regressions without re-running saucedemo's intentional per-user bugs three times. The result is 62 test instances total rather than ~120 for a full per-user-per-browser matrix.

See [ADR-0002](adr/0002-multi-user-via-projects-storage-state.md) for the multi-user pattern rationale and [ADR-0004](adr/0004-cross-browser-smoke-pattern.md) for the cross-browser smoke decision.

## CI Workflow Overview

The workflow lives at `.github/workflows/test.yml`. It runs a single job named `Playwright matrix` on `ubuntu-latest` with a `timeout-minutes: 15` cap.

**Triggers:** `push` to `main` and `pull_request` targeting `main`. A `concurrency` block keyed on `${{ github.ref }}` cancels any queued or running job for the same ref when a new run starts, preventing stacked runs from wasting minutes.

**Dependency installation:**

1. `actions/setup-node@v4` with `cache: 'npm'` installs Node 22 and restores the npm cache keyed on `package-lock.json`.
2. `npm ci` installs exact dependency versions from the lockfile.
3. `actions/cache@v4` restores the Playwright browser cache at `~/.cache/ms-playwright` keyed on `package-lock.json` hash. On a cache miss the next step downloads all three engines.
4. `npx playwright install --with-deps chromium firefox webkit` installs all three browser engines plus their Linux system dependencies. The `--with-deps` flag handles system packages (e.g., `libglib2.0`, `libnss3`) that Playwright requires on Ubuntu.

**Test run:** `npm test` executes the full 9-project matrix. The `SAUCEDEMO_PASSWORD` secret is supplied via `${{ secrets.SAUCEDEMO_PASSWORD }}`. The `CI: 'true'` environment variable activates retries (2) and the fixed worker count (4) in `playwright.config.ts`.

**Artifacts:** The HTML report (`playwright-report/`) is uploaded as an artifact only when the job fails (`if: failure()`), with a 7-day retention period. Successful runs produce no artifact вЂ” the HTML is only needed for failure diagnosis.
